## Epic Review

### Delivery Summary

- Story completion: 5/5 done.
- Retrospective precondition: satisfied (all Epic 1 stories done).
- Quality gate pattern: story check scripts and lint were executed and passed for each story.
- Architectural alignment trend: improved progressively from Story 1.2 onward as feature orchestration moved from route-level logic toward feature hooks/infrastructure boundaries.

### What Went Well

- Foundation stability was established early via starter-template baseline and consistent project structure.
- Local-first persistence patterns remained consistent across setup, profiles, parent-protection, and summary work.
- Parent-protected mutation boundaries were strengthened during review and patch cycles.
- The team handled review findings quickly with concrete patches instead of deferring.
- Story artifacts were kept up to date, making implementation and review traceable.

### Challenges and Friction

- Early stories had UI-level gating that needed to be hardened at mutation boundaries.
- Multiple review cycles identified that static grep checks were too weak and required behavior-oriented assertions.
- Profile and setup reset semantics needed explicit handling to avoid state confusion across restarts.
- Weekly summary source-of-truth required a decision and refactor to canonical local derivation.

### Cross-Story Patterns

1. Recurrent review theme: boundary enforcement.
- Mutation protections needed to be enforced in command paths, not only in UI controls.

2. Recurrent review theme: test depth.
- Validation scripts repeatedly evolved from token checks to stronger behavior-oriented assertions.

3. Recurrent review theme: deterministic data handling.
- Drift-prone or duplicate-source logic was corrected toward deterministic local records.

4. Positive pattern: architectural convergence.
- Changes trended toward clearer feature layering and better separation of route/UI from domain/infrastructure logic.

### Technical Debt and Risks

- Medium risk: canonical check-in ownership overlap.
  - Story 1.5 introduced `daily_checkins` for canonical weekly derivation before Epic 2 formalizes check-in/streak engine boundaries.
  - Risk: Epic 2 may introduce parallel representations unless schema ownership is explicitly unified.

- Medium risk: test strategy still script-heavy.
  - Story checks improved, but most acceptance validation remains shell-script assertions rather than runtime integration tests.

- Low risk: parent verification policy.
  - Current convenience PIN policy is acceptable for MVP pilot, but it remains a deliberate security tradeoff to revisit in later epics.

### Key Lessons Learned

- Enforce protected behavior at the command/mutation boundary first; UI gating is not sufficient.
- Decide canonical data ownership early when a story creates cross-epic foundational schema.
- Promote behavior-oriented checks sooner to reduce late review churn.
- Keep state-reset semantics explicit for restart-safe flows.
